# Project Onboarding Guide

当用户希望“快速上手一个已有项目”、“理解当前项目代码结构”、“为项目生成阅读教程/学习文档/章节讲解”时，使用这个 skill。

你的目标是帮助开发者逐步建立对项目的整体理解，并将结果沉淀到 `docs/tutorial/project-onboarding-guide` 目录下，优先复用已有大纲和章节内容。

## 总体原则

1. 每次启动本 skill 时，先检查 `docs/tutorial/project-onboarding-guide` 是否存在，以及其中是否已经有：
   - 项目理解大纲
   - 已生成的章节文档
2. 如果已存在相关文档，后续工作必须基于这些现有文档继续，而不是重新开始。
3. 如果用户没有提出额外要求，默认行为是：
   - 在 `docs/tutorial/project-onboarding-guide` 下创建一个“如何理解该项目”的章节大纲。
4. 在后续多轮对话中，用户可以：
   - 指定一个或多个章节，让你生成详细讲解内容；
   - 要求生成全部章节的讲解内容。
5. 每次生成具体章节内容时，必须先使用子 Agent 阅读项目代码，再由你整合结果并写入对应章节文件。
6. 内容应服务于“帮助开发者理解项目”，而不是泛泛介绍；讲解应尽量锚定实际代码、目录、模块、调用链和关键设计。

## 工作流程

### 第一步：检查现有文档状态

优先检查以下内容：

- `docs/tutorial/`
- `docs/tutorial/project-onboarding-guide/`
- 该目录下是否已有大纲文件，例如：
  - `docs/tutorial/project-onboarding-guide/README.md`
  - `docs/tutorial/project-onboarding-guide/outline.md`
  - `docs/tutorial/project-onboarding-guide/index.md`
  - 其他能明显看出是项目理解大纲的文件
- 该目录下是否已有章节文件

如果 `docs/tutorial` 不存在，则创建它。

如果 `docs/tutorial/project-onboarding-guide` 不存在，则创建它。

如果已有大纲：
- 读取并总结现有大纲结构；
- 在此基础上继续生成缺失章节或补充内容；
- 不要无故改写用户已有内容。

如果没有大纲：
- 基于项目实际结构创建一个新的理解大纲。

### 第二步：当用户没有明确要求时的默认行为

默认生成一个项目理解大纲，放在 `docs/tutorial/project-onboarding-guide/README.md`（优先）或 `docs/tutorial/project-onboarding-guide/outline.md`。

这个大纲至少应覆盖以下方面（按项目实际情况裁剪）：

1. 项目目标与业务定位
2. 运行方式与开发环境
3. 顶层目录结构
4. 应用入口与启动流程
5. 路由/页面/主流程
6. 核心模块划分
7. 数据模型与状态管理
8. 外部依赖与关键基础设施
9. 关键业务流程
10. 配置、构建与部署
11. 测试结构（如果有）
12. 建议的阅读顺序

输出的大纲应明确说明：
- 每一章为什么要读；
- 推荐阅读哪些目录、文件、模块或符号；
- 章节之间的依赖关系或推荐顺序。

### 第三步：当用户指定生成某些章节时

如果用户要求生成某一章或某几章：

1. 先读取现有大纲，确认章节名称、范围和上下文。
2. 如有必要，将用户的描述映射到最接近的大纲章节名称。
3. 对每个待生成章节，必须调用子 Agent 执行代码阅读任务。
a. 子 Agent 的任务应包含：
   - 章节名称；
   - 需要重点回答的问题；
   - 需要输出的代码证据（目录、文件、函数、类、调用关系、配置项等）；
   - 要求给出清晰的讲解结构，而不是只列文件。
b. 如果要生成多个章节，优先并行启动多个子 Agent，但要避免多个子 Agent 编辑同一个文件；子 Agent 只负责分析和撰写建议内容，由你统一落盘。
4. 你负责汇总子 Agent 的结果，整理成最终章节文档。
5. 将章节内容写入 `docs/tutorial/project-onboarding-guide` 下合适的文件名中。文件命名建议：
   - `01-project-overview.md`
   - `02-startup-and-entry.md`
   - `03-directory-structure.md`
   - 或与现有命名风格保持一致。

### 第四步：当用户要求生成全部章节时

如果用户要求生成整个项目的完整教程：

1. 先读取现有大纲；若没有则先生成大纲。
2. 按大纲拆分章节。
3. 为每个章节调用子 Agent 阅读代码并产出章节草稿或分析结果。
4. 由你统一整合、去重、修正术语，并写入 `docs/tutorial/project-onboarding-guide`。
5. 在内容很多时，优先保证每章都有：
   - 本章目标
   - 相关代码位置
   - 核心机制说明
   - 推荐阅读路径
   - 必要时补充调用链/数据流说明

## 章节内容写作规范

每个章节尽量采用如下结构（可按实际情况调整）：

```md
# 章节标题

## 本章要解决的问题

## 先看哪些文件

## 这一部分在项目中的职责

## 核心代码如何工作

## 关键类型 / 函数 / 类 / 配置

## 典型调用链或执行流程

## 阅读这一章后你应该理解什么

## 下一步建议阅读
```

写作要求：

- 内容必须基于仓库中的真实代码与文件结构。
- 解释优先具体，避免空泛表述。
- 应明确引用实际路径、模块名、函数名、类名、配置项。
- 如果某些信息无法从仓库中确认，要明确说明这是推断。
- 不要编造不存在的架构、流程或模块。
- 尽量帮助新开发者建立“从哪开始看、为什么这么设计、代码之间如何关联”的认知。

## 目录约定

本 skill 的所有产出文档都必须统一放在：

- `docs/tutorial/project-onboarding-guide/`

这意味着：

- 项目理解大纲默认放在 `docs/tutorial/project-onboarding-guide/README.md`；
- 章节文档默认也放在这个目录下；
- 不要将本 skill 的输出直接散落到 `docs/tutorial/` 根目录，除非用户明确要求；
- 如果 `docs/tutorial/` 已有其他教程或其他 skill 的文档，应与本 skill 的目录隔离。

## 关于大纲文件的建议格式

大纲文件建议包含：

- 项目理解目标
- 建议阅读顺序
- 章节列表
- 每章对应的重点目录/文件
- 每章预计回答的问题

可采用如下风格：

```md
# 项目理解指南

## 如何使用这份指南

## 推荐阅读顺序

## 章节目录

### 1. 项目概览
- 为什么读：
- 重点阅读：
- 你将理解：

### 2. 启动流程与入口
- 为什么读：
- 重点阅读：
- 你将理解：
```

## 与已有内容协同

如果 `docs/tutorial/project-onboarding-guide` 下已经存在：

- 大纲：优先沿用其章节编号、命名和结构；
- 部分章节：避免覆盖，除非用户明确要求重写；
- 风格不统一时：在保持原意的前提下做最小必要调整。

如果发现已有章节与代码不一致：
- 在更新时说明你是根据当前代码进行了校正；
- 不要静默大改用户文档。

## 执行细则

- 先做代码与文档状态检查，再决定是生成大纲、补章节，还是批量生成全部内容。
- 在生成章节正文之前，必须使用子 Agent 做代码阅读分析。
- 你可以自行编辑/创建文档文件；但子 Agent 只负责分析，不负责直接修改 `docs/tutorial/project-onboarding-guide`。
- 若项目规模较大，优先生成结构清晰、可持续扩展的文档，而不是一次性写成冗长大文。
- 如果仓库内容过少，仍应生成适配当前仓库规模的最小可用大纲，并如实说明信息有限。

## 交付结果

完成工作后，向用户简要说明：

- 检查到了哪些已有教程文件；
- 新增或更新了哪些 `docs/tutorial/project-onboarding-guide` 文件；
- 如果生成了章节，是基于哪些代码区域分析得出的；
- 下一步用户可以指定哪些章节继续展开。
